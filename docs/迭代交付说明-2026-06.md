## 一、我们先对齐：项目在解决什么问题

企盟不是单纯「通讯录 + 列表」，而是一套 **资源链接与变现** 的演示产品：

- 人：名片、好友、活动圈（同场认识但未加好友的人）
- 事：任务发布/承接/验收、活动报名
- 场：空间站场地预约
- 钱：合伙人会费、分润、贡献积分（链接者变现叙事）

**本轮目标**（你和我的共识）可以概括成两句话：

1. **从用户角度**：主流程能跑通，该点的地方能点、该看的信息能看懂，不要「有后端没入口」或「点了没反应」。  
2. **从交付角度**：代码可以提交、文档能交给下一个人/老板看懂；并为 **真实微信支付 + 上线** 铺好轨道，而不是永远停在「演示按钮」。

## 二、上线前用户会卡在哪（问题从哪来）

在改代码之前，实际走一遍小程序，用户最容易有「不好用 / 像没做完」的感受，主要来自这几类：

| 用户感受 | 背后原因（通俗说） |
|----------|-------------------|
| 「我的 → 记录」点了没反应 | 程序里写了「点开详情」的逻辑，但**页面上没有弹窗**，等于点了白点 |
| 「还要跑去任务广场看」 | 记录里只有跳转按钮，**没有在本页看完基本信息**，和「查历史」的预期不符 |
| 「记录排版乱」 | 活动日期、标签挤在一行；任务缺状态；和通讯录、任务 Tab 的卡片风格不统一 |
| 「支付像假的」 | 合伙人升级只有「模拟支付」，**没有接微信商户**（这是上线硬门槛，不是小 bug） |
| 「分享名片会不会坏」 | 分享接口写了，但**路由没挂到主程序**，真机会 404（已修） |
| 「文档和真机不一致」 | 推进计划还写「没有分润 Tab」，**和当前产品差了一整版**，老板/新同事会对不上 |

这些不是「功能完全没有」，而是 **体验闭环断了** 或 **文档与代码脱节**，演示时很容易被问住。

## 三、本轮具体改了什么（按用户路径讲）

下面按 **用户会怎么走** 来说，每项都写：**改了什么 → 为什么这么改**。

### 3.1 我的 → 记录（你重点提的需求）

**用户故事**：我在「我的」里想回顾自己报过的活动、接过的任务，不用切到任务广场翻列表。

**之前**：

- 列表能显示，但点击后没有详情界面（代码有 `openTaskDetail`，界面没做）。
- 弹窗方案里还放了「去任务广场查看」，用户会觉得多绕一步。

**现在**：

1. **点击活动/任务卡片 → 当前页弹窗看详情**  
   - 活动：时间、地点、报名情况、主办方、简介等。  
   - 任务：发布方、截止、名额、积分、状态、描述等。  
2. **去掉「去任务广场」**，只保留「关闭」——符合「记录 = 查阅档案」的定位。  
3. **列表样式重做**：标题一行、标签一行、摘要一行，右侧有 `›`，和 App 其它列表一致。  
4. **空状态**：没有记录时明确提示「暂无活动/任务记录」。

**设计考虑**：「记录」不承担操作（录用、交付、验收仍在 **任务&活动** Tab），避免一个页面承担两套心智；若以后要加「去处理」再单独加按钮即可。

---

### 3.2 合伙人支付（为真上线铺路）

**用户故事**：我在「我的 → 钱包 → 升级合伙人」里付钱，希望是真微信支付；演示环境可以先用模拟。

**之前**：只有 `mock-pay`，点完直接改数据库状态，无法对接商户号。

**现在（工程分层，不是一步到位真收钱）**：

| 层级 | 做了什么 |
|------|----------|
| 用户侧 | 按钮改为「确认支付」：先向服务器要预下单；能真付则调 `wx.requestPayment`；不能则走演示 mock |
| 服务器 | 统一「付成功以后干什么」：`fulfill_membership_order`（改会员状态、分润记账、贡献积分）——**mock 和微信回调共用**，避免两套逻辑以后不一致 |
| 配置 | `.env.example` +《上线与微信支付指南》：商户号、证书、回调 URL 怎么填 |
| 安全 | `APP_ENV=prod` 时关闭 mock，避免上线后还能「假支付」 |

**还没做完（要说清楚给老板）**：微信 V3 **真正下单、回调验签** 仍是骨架（配好商户号后要 1～2 周开发通联）。演示环境**行为不变**，仍可一键模拟支付走通分润故事。

---

### 3.3 名片分享与数据（隐性阻断）

**用户故事**：分享名片、带分享码注册。

**问题**：`card-shares` 接口文件存在，但**没注册到主应用**，分享/追踪会失败；分享计数写了 `total_shares` 字段但表里没有，会报错。

**已修**：挂上路由；用户表增加 `total_shares` 及数据库迁移。

## 四、没动但你要心里有数的「体验仍偏弱」

诚实列给老板，避免过度承诺：

| 模块 | 现状 | 用户感受 |
|------|------|----------|
| 钱包「转让」「流水」 | 按钮在，未接功能 | 像占位 |
| 我的股东权益 | 接口有，无独立页 | 空间站里能看部分，缺「我的权益」入口 |
| 活动圈 | 功能完整 | 需甲/乙同场报名才测得出，文档已写步骤 |
| 真实微信支付 | 骨架 | 演示 OK，上线前必须通联 |
| 月结 / 运营后台 | 未做 | 分润只能「确认」，不能系统「已发放」 |
| 生产环境 | 仍默认 SQLite 本地库 | 上线要 PostgreSQL + HTTPS |

这些属于 **下一阶段**，不是本轮「骗人说是成品」。

## 七、建议你给老板口头汇报的 3 分钟版

> 我们这版企盟小程序，**核心业务已经能在本地和真机上跑通**：通讯录、任务活动、空间站、分润和贡献积分、我的档案和记录。  
>  
> 这一轮主要解决的是 **「体验断点」**：比如「我的记录」以前点了没详情，现在可以在本页看完活动/任务信息；文档和真实功能对齐，方便团队交接。  
>  
> 同时我们为 **真微信支付和上线** 拆了模块：支付成功后的分润、会员状态只维护一套逻辑，避免以后改两遍；并写了上线 checklist。  
>  
> **还没做的是**：微信商户真实收款、生产数据库部署、月结和运营后台——这需要企业资质和大约 3～4 周，和演示版是两条线。  
>  
> 建议下一步：先用双账号把演示路径验收一遍给甲方；并行申请商户号，排上线开发。
